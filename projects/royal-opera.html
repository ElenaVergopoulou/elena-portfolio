Overview

This client project focused on evaluating the accessibility of the Royal Opera House website, with a particular emphasis on inclusive design for users with visual, auditory, motor, and cognitive impairments. By applying WCAG 2.2 Level A and AA guidelines, I aimed to uncover critical usability barriers and recommend practical interventions that enhance access and usability for a broader audience.

I examined multiple journeys across desktop and mobile views, identifying 10 high-impact issues ranging from poor focus visibility to the absence of alt text and missing skip links. The goal was to go beyond compliance, considering real-world accessibility needs and proposing realistic, actionable changes.

Homepage of the Royal Opera House website ‚Äì part of the evaluation for accessibility across screen readers, keyboard navigation, and layout responsiveness.

Project Goals

  1. Evaluate the Royal Ballet & Opera website against WCAG 2.2 A and AA standards: Use accessibility guidelines to assess compliance and real-world usability for users with disabilities.
  2. Identify common and critical accessibility issues affecting diverse user groups: Focus on visual, auditory, motor, and cognitive impairments to ensure broad inclusivity.
  3. Use a combination of automated tools, screen readers, and manual inspection: Cross-validate findings to uncover both technical and experiential barriers to access.
  4. Translate accessibility findings into clear, actionable recommendations: Provide realistic suggestions that could be adopted by the Royal Opera House digital team.
  5. Raise awareness about inclusive design practices in a real-world cultural context: Use this project to demonstrate how digital accessibility intersects with public service and cultural engagement.

My Role

I was responsible for conducting a comprehensive accessibility evaluation and delivering clear, actionable outcomes for the client. My contributions included:

  • Conducting an accessibility audit across desktop and mobile views of multiple key pages
  • Running tests using tools like WAVE, Lighthouse, Silktide, PEAT, and NVDA
  • Performing manual HTML inspections to assess aria labels, heading structure, alt text, and keyboard accessibility
  • Mapping all findings to WCAG 2.2 success criteria, with severity ratings and user impact explanations
  • Writing a detailed, client-ready presentation and preparing a supporting script to guide the walkthrough
  • Designing inclusive solutions and providing rationale for prioritization and implementation feasibility

Methodology

This project followed a structured, tool-supported accessibility evaluation framework across multiple devices and key pages.

Automated Testing

  • Used WAVE and Lighthouse to detect structural issues, contrast failures, missing labels, and focus visibility problems

Manual Code Inspection

  • Inspected HTML and CSS manually to assess semantic markup, proper ARIA roles, heading hierarchy, and keyboard trap risks
  • Checked for missing alt attributes, improper use of aria-hidden, and link/button labeling

Assistive Technology Testing

  • Used NVDA screen reader to assess screen reader compatibility, logical reading order, and keyboard accessibility

Simulation Tools

  • Used Silktide to simulate dyslexia, color blindness, and visual impairments to evaluate layout clarity and content readability
  • Used PEAT to test for seizure-inducing flashing elements

All issues were categorized by WCAG 2.2 success criteria, with severity levels and real-world impact noted. Results were organized into a presentation and summary table for stakeholder delivery.

Findings

The accessibility evaluation revealed several critical issues that could significantly hinder usability for users with visual, motor, auditory, or cognitive impairments. These findings were mapped to WCAG 2.2 A and AA criteria, and categorized by severity and user impact.

  • Zooming limitations: The layout broke when zoomed in‚Äîmenus and page elements disappeared or overlapped. This affects users with low vision, dyslexia, or those on small screens.
  • Poor focus indicators: Focus outlines were low contrast or completely invisible (e.g., black on black), making it nearly impossible for keyboard-only users to navigate.
  • Missing ‚ÄúSkip to Content‚Äù link: Users had to tab through repeated header content on every page, increasing physical and cognitive load.
  • Unlabeled search icon and form: The search icon read as ‚Äúicon‚Äù and the text field lacked a programmatic label‚Äîconfusing for screen reader users and those using voice control.
  • Missing video captions: Videos lacked captions and transcripts, making them inaccessible to deaf or hard-of-hearing users and difficult to follow in low-audio environments.
  • Identical link text with different destinations: Some links used the same label but led to different pages, which disrupts screen reader navigation and increases ambiguity.
  • Improper heading structure: Skipped heading levels and empty heading tags disrupted the semantic reading flow, affecting screen reader comprehension.
  • Missing alt text: While most images were labeled, some meaningful content lacked descriptive alt attributes, making it invisible to screen readers.
  • No lang attribute: The absence of a language tag caused incorrect pronunciation by screen readers and hindered accessibility for multilingual users.

Screen reader simulation: Navigation lacked programmatic structure, resulting in disorganized reading order.

Repetitive link text: Multiple ‚ÄúMore info‚Äù links lacked contextual labels, causing confusion for screen reader users.

Heading structure: Skipped heading levels and inconsistent hierarchy disrupted screen reader navigation.

Missing language attribute: The lang attribute was not defined, affecting pronunciation accuracy in screen readers.

Recommendations

Based on the accessibility evaluation, the following improvements were proposed to address the most impactful barriers for users with disabilities. Each recommendation is aligned with WCAG 2.2 standards and grounded in inclusive design principles.

Add descriptive alternative text

Ensure all non-decorative images include meaningful alt attributes to support screen reader interpretation and visual context for users with low vision.

Improve color contrast and visual clarity

Adjust foreground/background combinations, especially in navigation menus and CTAs, to meet minimum contrast ratios (4.5:1 for normal text).

Introduce consistent and visible focus styles

Apply custom keyboard focus indicators across all interactive elements to support users navigating via keyboard or assistive devices.

Add skip-to-content links

Implement skip links at the start of each page to allow users to bypass repetitive header content and reach main content quickly.

Reorganize heading levels

Follow a logical and nested heading structure (e.g., H1 ‚Üí H2 ‚Üí H3) to improve page hierarchy comprehension for screen reader users.

Label all form inputs and interactive components

Use programmatically associated labels (label for or aria-label) for all inputs, buttons, and icons to improve screen reader usability.

Specify the language attribute in the HTML

Add lang="en" to the HTML tag to ensure proper pronunciation and localization for screen readers.

These recommendations are practical for implementation within existing CMS or development frameworks. In a real-world client context, they could be supported with annotated screenshots and validation via stakeholder feedback.

Outcome

The final presentation delivered to the client outlined ten core accessibility issues, each mapped to specific WCAG 2.2 A and AA criteria. It clearly documented how these issues impact users with visual, motor, auditory, and cognitive impairments.

  • Structured analysis aligned with accessibility standards
  • Realistic, low-disruption solutions tailored to the current site architecture
  • Use of simulations and screen reader testing to illustrate impact
  • Clear prioritization based on user impact, not just technical compliance

The project demonstrated how small, strategic changes‚Äîlike consistent focus indicators or proper alt text‚Äîcan meaningfully improve access for underserved users. It also reinforced the importance of embedding inclusive thinking into every stage of digital product development.

Reflection

This project deepened my understanding of accessibility as an essential part of the design process. It strengthened my ability to evaluate digital experiences from the perspective of users facing real-world barriers, and pushed me to communicate findings clearly and constructively.

What Worked Well

  • Alignment with WCAG: Mapping all findings to WCAG 2.2 A/AA standards helped communicate severity and actionability to stakeholders.
  • Empathy through simulation: Tools like Silktide and PEAT helped highlight issues beyond the obvious, fostering more thoughtful recommendations.

What I‚Äôd Improve
